Описание интерфейса и протоколов

Вот детальная техническая спецификация протоколов Ring‑Ring и архитектуры интерфейса пользователя.
Часть A: Протоколы и криптография

Это техническая основа для разработчиков и аудиторов.

1. Идентификация и аутентификация:

    Первичный идентификатор: Пара ключей Ed25519, генерируемая локально во время первичной инициализации. Публичный ключ служит уникальным идентификатором узла в сети.

    Привязка к телефонному номеру:

        Процесс: Клиент отправляет номер на сервер верификации, получает OTP-код. После этого клиент подписывает свой публичный ключ этим кодом. Запись blake2b(номер_телефона) -> подписанный_публичный_ключ публикуется в DHT. Сервер верификации не хранит никаких данных.

        Поиск контакта: Клиент вычисляет blake2b(введённого_номера), делает запрос в DHT, проверяет подпись и получает публичный ключ.

        Конфиденциальность: По умолчанию номер виден только в хэшированном виде в DHT. Механизмы профиля позволяют скрыть его от других пользователей.

    Авторизация сессии: После первичной настройки устройство аутентифицируется в сети с помощью цифровой подписи на основе своего приватного ключа.

2. Криптографические примитивы и их использование:

    Шифрование канала (транспортное): Noise Protocol Framework. Выбран паттерн XX (для максимальной аутентификации и сокрытия идентичности). Криптографическая стойкость определяется использованием эфемерных ключей Curve25519 для DH и алгоритма AES-256-GCM для шифрования.

    Шифрование сообщений (E2EE): Осуществляется в рамках сессии Noise. Для реализации Perfect Forward Secrecy и асинхронной доставки офлайн-сообщений (очередь в DHT) может быть использована гибридная схема: для каждого сообщения генерируется эфемерный ключ (EPK), сообщение шифруется с помощью AES-256-GCM на ключе, полученном через DH между EPK и постоянным публичным ключом получателя. Это стандартный механизм, реализуемый во многих современных протоколах.

    Хэширование: Blake2b (для адресов, хэшей ключей, отпечатков).

    Ключи и их длина (Относительно пункта 2 вашего запроса):

        AES: Для симметричного шифрования используется AES-256-GCM. Это стандарт NIST. Длина ключа — 256 бит. Длина блока — 128 бит. AES-128 и AES-192 не используются.
        RSA: НЕ РЕКОМЕНДУЕТСЯ К ИСПОЛЬЗОВАНИЮ в качестве основного криптографического примитива в протоколе. Рекомендуемые альтернативы — кривые X25519 (для DH) и Ed25519 (для подписей). Если в каких-либо нестандартных модулях (например, в устаревших частях кода регистрации, написанных для обратной совместимости) и используется RSA, то минимальная длина ключа — 2048 бит. Длина в 512 или 1024 бита считается криптографически небезопасной и НЕДОПУСТИМОЙ в любых защищённых протоколах на момент 2026 года.

3. Сетевой протокол:

    Транспорт: UDP. Для соединений, требующих надёжности (чат, передача файлов), поверх UDP реализуется простая фрагментация и подтверждения (ACK/NACK). Для голоса/видео используется RTP поверх UDP.

    Обнаружение узлов (Peer Discovery):

        Локальная сеть: mDNS.

        Глобальная сеть: Реализованный в libp2p протокол Kademlia DHT. Узлы публикуют в DHT запись: публичный_ключ -> [мультиадрес].

    Установление соединения и NAT traversal (Hole punching): Используется механизм Circuit Relay из libp2p, когда оба узла находятся за симметричными NAT.

4. Офлайн-сообщения:
Отправитель шифрует сообщение, разделяет его на блоки и помещает в DHT по ключу /offline/публичный_ключ_получателя/ID_сообщения. При появлении в сети получатель сканирует DHT по своему ключу и забирает все ожидающие сообщения.
Часть B: Архитектура интерфейса пользователя (UI)

Цель: Создать интуитивно понятный, быстрый и конфиденциальный интерфейс, вдохновляясь лучшими решениями, но с учётом строгих ограничений P2P архитектуры. Копирование исходного кода UI Telegram прямо запрещено его лицензией GPLv3: вы обязаны открыть исходный код любых производных работ, но при этом Telegram сохраняет авторские права на оригинальный код.

Основные экраны:

    Экран входа/регистрации: Простой ввод номера телефона и одноразового кода. Под капотом — локальная генерация ключей, как описано в Части A.

    Список чатов (Inbox): Список последних диалогов с превью сообщений. Сортировка по времени последней активности.

    Экран чата: Область сообщений (текст, эмодзи, ссылки), поле ввода, кнопки для отправки файлов и начала звонка.

    Экран контактов: Список контактов (те, с кем есть общий ключ) и поиск новых по номеру телефона или публичному ключу.

    Настройки профиля:

        Управление видимостью номера (для всех/только для контактов/никому).

        Управление подключенными устройствами.

        Настройки конфиденциальности (кто может видеть статус "онлайн").

Влияние P2P на UI:

    Статусы "онлайн" и "печатает": Эти статусы работают только при прямом P2P-соединении. Офлайн-метки могут обновляться с задержкой.

    Поиск контактов: Поиск по номеру телефона может быть немного медленнее из-за запросов к DHT.

    Доставка сообщений: UI должен явно показывать статус: "отправляется..." → "доставлено" (есть соединение) → "будет доставлено позже" (сохранено в DHT).

Кроссплатформенность:

    Фреймворк: Flutter для единого кода под iOS, Android, Windows, macOS и Linux.

    Бэкенд: Библиотека на Rust, интегрированная с Flutter через Dart FFI.
